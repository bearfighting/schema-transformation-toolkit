# Release Process

This repository now supports a minimal release path for GitHub tags and GitHub Releases without publishing any packages to npm.

That is the intended current baseline.
Until the parser and generator surface is broader and the consumer contract has settled further, releases should be treated as tagged engine snapshots plus attached package tarballs rather than as public npm publication events.

## Current Goals

The current release path is meant to provide:

- consistent workspace versioning
- annotated git tags
- GitHub Releases with autogenerated notes
- attached package tarballs for the current workspace packages

It is intentionally not meant to provide:

- npm publication
- a public package-registry support promise
- independent per-package versioning

## Versioning Model

The repository currently uses one shared version across:

- the root `package.json`
- every workspace package under `packages/`

Recommended version forms:

- prerelease: `0.1.1-alpha.0`
- stable: `0.1.1`

Recommended tag form:

- `v0.1.1-alpha.0`
- `v0.1.1`

For now, keep the whole workspace on one version.
That keeps release reasoning simple while the package surface is still evolving together.

## Release Scripts

The root workspace now exposes:

- `pnpm release:check`
- `pnpm release:version <version>`
- `pnpm release:tag`
- `pnpm release:pack`

What they do:

- `release:check`: verifies version consistency and, by default, expects a clean worktree and a not-yet-existing release tag
- `release:version`: updates the root and all workspace package versions together
- `release:tag`: creates an annotated `v<version>` git tag from the current shared version
- `release:pack`: creates release tarballs and a `release-manifest.json` file in `release-artifacts/`

## Recommended Local Release Flow

For a normal release:

1. run `pnpm check`
2. choose the next version, for example `0.1.1-alpha.0`
3. run `pnpm release:version 0.1.1-alpha.0`
4. review the version-only diff and commit it
5. run `pnpm release:check`
6. run `pnpm release:tag`
7. push the commit and tag:
   - `git push origin main`
   - `git push origin v0.1.1-alpha.0`

After the tag push, GitHub Actions will:

- install dependencies
- verify the release state against the pushed tag
- run repository checks
- build the workspace
- pack workspace package tarballs
- create a GitHub Release with autogenerated notes

## GitHub Release Behavior

The release workflow lives in [../../.github/workflows/release.yml](../../.github/workflows/release.yml).

It is tag-driven:

- pushing any tag matching `v*` triggers the workflow

The current workflow uses GitHub autogenerated release notes.
That autogenerated GitHub Release body should be treated as the canonical release note for now.

If a release needs a short manual preface, edit the GitHub Release after creation rather than maintaining a separate hand-written changelog during this phase.

## Attached Release Artifacts

The release workflow attaches:

- one tarball per workspace package
- one `release-manifest.json` file

The manifest records:

- tag
- version
- generated timestamp
- package names
- package versions
- tarball filenames

This is the current non-npm distribution path.

## Downstream Consumption Guidance

Until npm publication is introduced, downstream consumers should prefer one of these models:

1. depend on a tagged repository snapshot inside a larger internal workspace
2. consume the release tarballs from a matching GitHub Release
3. vendor the tagged source snapshot directly when a monorepo-style integration is simpler

The important rule is consistency:

- use one release tag across all consumed workspace packages
- do not mix tarballs from different release tags

## What Still Remains Later

This first pass does not remove future release work.
Likely later follow-up includes:

- a public changelog policy
- npm publication
- package visibility changes from `private: true`
- possible package-level publish filtering
- stronger automation around release PRs or prerelease channels

Those should wait until the package surface is more stable.
